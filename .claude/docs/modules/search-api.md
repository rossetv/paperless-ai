# Module: search-api

## Purpose

The HTTP/ASGI surface of the search server: one uvicorn process serving the `/api/*` REST routers, the `/mcp` streamable-HTTP MCP endpoint (5 tools), and the built React SPA with a deep-link catch-all. It owns authentication (server-side sessions behind an opaque hashed `search_session` cookie; `sk-pls-` bearer API keys with `api`/`mcp`/`admin` scopes), RBAC (`readonly` < `member` < `admin`), the Pydantic wire boundary (`search/wire/`), config hot-reload, per-key spend quota and security headers. It never runs the search pipeline itself — it wires `SearchCore` and offloads blocking work to the threadpool.

Entrypoint: console script `paperless-search-server` → `search.api:main` (`pyproject.toml`), which builds the app with `search.api.create_app(settings, *, core=None, store_reader=None, app_db_path=None)` and runs uvicorn with `proxy_headers=True`, `forwarded_allow_ips=settings.SEARCH_FORWARDED_ALLOW_IPS`. Tests call `create_app(...)` directly with injected stubs and drive it with `TestClient`.

## Invariants

- **Two credential kinds only.** A `search_session` cookie (a human — bounded by role alone, never scope-limited) or `Authorization: Bearer sk-pls-...` (bounded by BOTH the key's scopes and the owner's *current* role). The legacy `SEARCH_API_KEY` shared secret is retired: a fresh install has zero programmatic access until a key is minted.
- **No secret is persisted or logged in the clear.** Session tokens and API keys are stored only as SHA-256 hashes (correct here — both are full-entropy random, so no slow KDF is needed); passwords use argon2id. Auth failures log only whether a header/cookie was present, never its value. `?reveal=true` logs the revealed *keys*, never the values.
- **Every credential check fails closed and returns a value rather than raising.** `resolve_session`, `resolve_api_key`, `verify_setup_token` and `authorise_role` (unknown role ranks -1) return None/False, so a hostile request can never trigger a 500.
- **One sqlite3 connection per request**, opened by `deps.get_app_db` and closed in its `finally`. `AppState` holds the app.db *path*, never a live connection (a `sqlite3.Connection` is not safe across FastAPI's request threads). Two deliberate exceptions: the stream worker and `spend_quota` each open their OWN short-lived connection, so a client disconnect closing the request connection cannot race them.
- **The search server NEVER writes index.db.** The indexer holds the exclusive writer flock. Reconcile and rebuild are *requested* by touching `reconcile.request` / `rebuild.request` beside index.db.
- **RBAC ranks readonly(0) < member(1) < admin(2)**, gated per route:

  | Gate | Routes |
  |------|--------|
  | public | healthz, setup, setup-status, login, stats-public |
  | readonly + `api` scope | search, stream, facets, stats, documents browse, PDF/thumb, recent-searches, taxonomy list, index status/activity/failed |
  | member + `api` scope | reconcile, document PATCH, reclassify, retranscribe, taxonomy create |
  | admin + `admin` scope | document DELETE, user CRUD, settings, index rebuild |
  | member + `admin` scope | API-key management |
  | any active session OR a key with `mcp` | `/mcp` |

- **Never zero admins, never self-foot-gun.** A user cannot delete/suspend/demote themselves, and the last *active* admin cannot be deleted/suspended/demoted. The guard read and the write share one `BEGIN IMMEDIATE`, so the invariant is race-free under concurrent admin edits.
- **`SEARCH_MAX_CONCURRENT` is ONE ceiling** across `/api/search`, `/api/search/stream` and the MCP tools — a single `LazySemaphore` built in `create_app` and injected into both surfaces, not 2N. A ceiling of 0 means unbounded (nullcontext), never `asyncio.Semaphore(0)`.
- **Configuration hot-loads with no restart.** `_resolve_search_core(app_db_path)` reads `config_version` per request (one cheap SELECT) and rebuilds the `SearchCore`, Settings, `setup_libraries` (OpenAI client holder), `llm_limiter` and the result cache only when the version moves. The rebuild is serialised by `_CORE_CACHE_LOCK`; `current_settings_with_version` takes one `BEGIN DEFERRED` snapshot so the cached version always describes the cached settings.
- **A `REINDEX_KEYS` save schedules the rebuild BEFORE the config write commits.** If the sentinel write fails the whole save rolls back (503) — a new `EMBEDDING_MODEL`/chunking value can never persist and hot-load without its wipe scheduled.
- **Query normalisation is one rule at both boundaries:** `search.wire.normalise_query` — max 4000 chars, trimmed, empty/whitespace-only rejected (422 over REST, a tool error over MCP).
- **Streaming search cannot fail with an HTTP status once the body has begun.** A budget breach becomes a `budget` error frame, a mid-rebuild index an `index_not_ready` frame, anything else an `internal` frame. The quota pre-check therefore runs *before* the body starts (so an over-quota key still gets a clean 429) and the usage record runs inside the worker.
- **Streamed bodies and per-request `PaperlessClient`s are always cleaned up.** `_safe_stream` closes the client in a `finally` covering drain, mid-stream error and the `GeneratorExit` Starlette raises on client disconnect; the NDJSON `frames()` generator `await`s the worker in a `finally` so a disconnect still reaps the thread.
- **Only `deep_search` bills the archive owner's LLM budget.** `keyword_search`, `fetch_documents` and `list_filters` pass `bills_llm=False`, make zero chat-LLM calls and skip the spend quota entirely. (`semantic_search` also makes no chat call but still routes through `_dispatch` — see Gotchas.)
- **The MCP tool result drops `stats.trace` before serialising** — the verbose per-phase reasoning is an SPA-only affordance; MCP clients get answer, sources, plan and the `cost` summary only.

## Gotchas

- `/mcp` is attached as an exact-path `Route("/mcp", mcp_app)` appended to `app.router.routes`, **not** `app.mount`. A `Mount` double-prefixes to `/mcp/mcp` and 405s the bare `/mcp` (the SPA catch-all's method mismatch shadows Starlette's slash-redirect). FastMCP's `streamable_http_path` must therefore stay at its default `/mcp`. Regression-tested in `tests/integration/test_mcp_mount.py`.
- An ASGI sub-app attached by `Route` or mount never has its lifespan run by the parent. `create_app`'s own `lifespan` explicitly runs `mcp_app.session_manager.run()`; without it every `/mcp` request fails with `RuntimeError: Task group is not initialized` → 500.
- **Route registration order is load-bearing:** `/mcp` Route → account → settings → api_key → document → index → api routers → `register_spa` LAST. The SPA catch-all is `GET /{full_path:path}`, so anything registered after it is shadowed for GETs. The catch-all also hard-refuses `api/`- and `mcp`-prefixed paths as a second line of defence.
- `FRONTEND_DIST` is resolved at *import* time of `search.api` (env var, else `<repo>/web/dist`). Tests exercising the SPA must set the env var and reload the module (see `tests/integration/test_accounts_legacy_and_spa.py`).
- `create_app` starts two `daemon=True` background threads — the heartbeat (`_start_search_heartbeat`, every 30 s) and, only when `PRICING_REFRESH_URL` is set, the price-refresh loop. Every test that builds the app spawns a heartbeat thread; with `PRICING_REFRESH_URL` unset (the prod default) no price thread starts and no network call is made.
- `_CORE_CACHE` is a process-global dict keyed by app.db path. Tests that build multiple apps against the same path must call `_reset_core_cache_for_test()` or they get a stale core.
- `SEARCH_FORWARDED_ALLOW_IPS` defaults to `"*"` — uvicorn trusts `X-Forwarded-For` / `X-Forwarded-Proto` from ANY peer. Only safe because the uvicorn port is reachable solely by the reverse proxy. If that port ever becomes directly reachable, an attacker can forge the audited `sessions.ip` and flip the cookie `Secure` flag. The IP-independent per-username login counter exists precisely because of this.
- `search.deps.require_mcp_scope` and `search.auth.api_key_caller` have **no production callers** (unit-tested only). The MCP surface is raw ASGI and does its own check in `_BearerAuthMiddleware._resolve_caller`; `deps.resolve_caller` calls `resolve_api_key` directly. Don't assume they are on a live path.
- `_BearerAuthMiddleware` sets the `mcp_asker` / `mcp_api_key_id` ContextVars, but `run_blocking` uses `run_in_executor`, which does **not** propagate contextvars to the worker thread. A tool body must read the ContextVar on the loop and capture it into the closure — `_dispatch` and `_run_tool` do exactly this; reading them inside `_build` would silently get the default (None).
- In `mcp_server._register_search_tools`, `semantic_search` goes through `_dispatch`, which hard-codes `bills_llm=True` — so it is quota-checked and records tokens even though `core.retrieve` makes zero chat-LLM calls (the recorded total is 0). The module docstring claims four free tools; only three actually skip the quota.
- The login throttle, the search-result cache and the core cache are all **process-local singletons**. The architecture assumes a single search-server process; a multi-instance deployment would silently get N× the intended lockout/quota leniency and inconsistent config-version caches. Same for the setup token, held only in memory — restarting before setup completes mints a fresh one.
- `SecurityHeadersMiddleware` is **additive** — it skips any header a handler already set. The PDF proxy exploits this to override `X-Frame-Options: DENY` → `SAMEORIGIN` and CSP `frame-ancestors 'none'` → `'self'` so the SPA can embed the PDF in a same-origin iframe. Making the middleware overwrite would break the in-app PDF viewer.
- The thumbnail proxy forwards the upstream content type only if it is in a 4-entry image allowlist; anything else (an `.html`/`.svg` stored in Paperless) is a 502. The PDF proxy never forwards the upstream type at all — it is pinned to `application/pdf`. Both are stored-XSS defences; do not 'fix' them by passing the upstream type through.
- `PATCH /api/api-keys/{id}` returns **404, not 403**, to a non-owner — even an admin — and `DELETE` returns 404 to a non-owning non-admin. Deliberate id-enumeration defence. Editing is owner-only; revoking is owner-or-admin. The asymmetry is intentional.
- `PUT /api/settings` rejects the literal mask `********` for any `SECRET_KEYS` value at the boundary — the frontend omits an unchanged secret, so a payload carrying the mask is a bug or an attack.
- `POST /api/settings/test-connection` injects a `__test_connection_placeholder__` sentinel for every missing secret so `build_settings` validates, then short-circuits on the sentinel **before** probing (`ok=False, detail="Not configured"`). The check is on the merged dict, not the built Settings, because `build_settings` nulls `OLLAMA_BASE_URL` when `LLM_PROVIDER=openai`.
- `/openapi.json`, `/docs` and `/redoc` are all disabled (`openapi_url=None`). Nothing first-party consumes the schema, and serving it would leak every route and model to anyone past the Access edge.
- The stream emits a bare `\n` every 15 s of silence (`_STREAM_KEEPALIVE_SECONDS`) so an idle Cloudflare/nginx hop does not kill the connection during a long synthesiser call. **Clients MUST skip blank lines.**
- FastMCP's DNS-rebinding Host/Origin check is explicitly disabled (`TransportSecuritySettings(enable_dns_rebinding_protection=False)`) — it auto-enables a localhost-only allowlist that 421s every real request behind the reverse proxy. The bearer-auth middleware, the SameSite=Strict cookie and CSP `connect-src 'self'` cover the threat model.

## Extension points

- **A new REST route** — add it to an existing router (or a new `build_*_router` factory) and register it in `create_app` **before** `register_spa`. Gate it with a `deps.require_*` dependency; never hand-roll an auth check.
- **A new MCP tool** — `@mcp.tool` inside `mcp_server._register_search_tools`, returning `await _run_tool(_build, bills_llm=...)`. Set `bills_llm=True` only if the tool actually spends the owner's chat-LLM budget.
- **A new request/response shape** — a Pydantic model in `search/wire/<area>.py`, re-exported from `search/wire/__init__.py`. Pydantic must not leak past this package into the pipeline.
- **A new config key** — add it to `common.config` (`CONFIG_KEYS`, plus `SECRET_KEYS`/`REINDEX_KEYS` as applicable); the Settings GET/PUT surface iterates `CONFIG_KEYS`, so it appears with no route change. A key that invalidates the index must be in `REINDEX_KEYS`.
- **A new probe** for `POST /api/settings/test-connection` — add a `_probe_*` that returns a result rather than raising; the route must never 500.
